## Accurate Takeaways

*Plain statements validated from sources. No marketing language.*

1. `SNOWFLAKE.ACCOUNT_USAGE` provides account-level historical usage/metadata views with longer retention than Information Schema, but with non-zero data latency; Snowflake documents specific latencies per view and recommends using Snowflake database roles for least-privilege access. 
   - Source: Account Usage overview and role guidance. [Snowflake Docs: Account Usage](https://docs.snowflake.com/en/sql-reference/account-usage)

2. `SNOWFLAKE.ACCOUNT_USAGE.QUERY_HISTORY` can be queried for up to 365 days of query history and includes warehouse/user/role dimensions (with up to ~45 minutes latency). 
   - Source: [Snowflake Docs: QUERY_HISTORY view](https://docs.snowflake.com/en/sql-reference/account-usage/query_history)

3. `SNOWFLAKE.ACCOUNT_USAGE.QUERY_ATTRIBUTION_HISTORY` provides per-query *compute* cost attribution (credits) for queries executed on warehouses, but explicitly **excludes warehouse idle time**, and also excludes other cost categories such as storage, data transfer, cloud services, serverless features, and AI token costs. Snowflake also notes short-running queries (~<=100ms) may be excluded and that view latency can be up to ~8 hours.
   - Source: [Snowflake Docs: QUERY_ATTRIBUTION_HISTORY view](https://docs.snowflake.com/en/sql-reference/account-usage/query_attribution_history)

4. `WAREHOUSE_METERING_HISTORY` exists as an Information Schema **table function** that returns hourly credit usage for the last ~6 months; Snowflake notes that for longer periods or completeness, you should use the Account Usage view instead.
   - Source: [Snowflake Docs: WAREHOUSE_METERING_HISTORY table function](https://docs.snowflake.com/en/sql-reference/functions/warehouse_metering_history)

5. A practical, transparent reconciliation approach is to treat `WAREHOUSE_METERING_HISTORY` as “credits metered” at hourly grain and allocate those credits across queries and idle time windows in that hour; third-party practitioners report validating Snowflake’s per-query attribution outputs and sometimes observing mismatches, motivating an internally computed “metered hour allocation” model.
   - Source (practitioner perspective): [Greybeam: Snowflake cost per query + idle time attribution](https://blog.greybeam.ai/snowflake-cost-per-query/)

## Concrete Artifacts

### Artifact: Warehouse-hour reconciliation + idle credits (SQL draft)

Goal: produce a daily/hourly dataset for a FinOps Native App that reconciles:
- **Metered credits**: warehouse hourly credits (canonical)
- **Attributed credits**: sum of per-query compute attribution in same warehouse/hour
- **Idle credits**: residual = metered - attributed (bounded at >= 0)

```sql
-- Warehouse-hour reconciliation: metered vs attributed
-- NOTE: Validate exact column names for ACCOUNT_USAGE.WAREHOUSE_METERING_HISTORY in your account.
-- The QUERY_ATTRIBUTION_HISTORY semantics are documented to exclude idle time.
-- Sources:
-- - QUERY_ATTRIBUTION_HISTORY docs: https://docs.snowflake.com/en/sql-reference/account-usage/query_attribution_history
-- - QUERY_HISTORY docs: https://docs.snowflake.com/en/sql-reference/account-usage/query_history

SET start_ts = DATEADD('day', -14, CURRENT_TIMESTAMP());
SET end_ts   = CURRENT_TIMESTAMP();

WITH qa AS (
  SELECT
    warehouse_id,
    DATE_TRUNC('hour', start_time)          AS hour_start,
    SUM(credits_attributed_compute)         AS attributed_compute_credits,
    SUM(COALESCE(credits_used_query_acceleration, 0)) AS qa_credits
  FROM SNOWFLAKE.ACCOUNT_USAGE.QUERY_ATTRIBUTION_HISTORY
  WHERE start_time >= $start_ts
    AND start_time <  $end_ts
  GROUP BY 1,2
),

-- If you also want user/role chargeback, you can pull it from QUERY_HISTORY and join by query_id
-- (QUERY_ATTRIBUTION_HISTORY already includes USER_NAME + QUERY_TAG, but QUERY_HISTORY adds ROLE_NAME, etc.)

wmh AS (
  SELECT
    warehouse_id,
    start_time AS hour_start,
    credits_used_compute,
    credits_used_cloud_services
  FROM SNOWFLAKE.ACCOUNT_USAGE.WAREHOUSE_METERING_HISTORY
  WHERE start_time >= DATE_TRUNC('hour', $start_ts)
    AND start_time <  DATE_TRUNC('hour', $end_ts)
)

SELECT
  wmh.warehouse_id,
  wmh.hour_start,
  wmh.credits_used_compute                           AS metered_compute_credits,
  qa.attributed_compute_credits,
  GREATEST(wmh.credits_used_compute - COALESCE(qa.attributed_compute_credits, 0), 0)
    AS residual_idle_or_unattributed_compute_credits,
  wmh.credits_used_cloud_services                    AS metered_cloud_services_credits
FROM wmh
LEFT JOIN qa
  ON qa.warehouse_id = wmh.warehouse_id
 AND qa.hour_start   = wmh.hour_start
ORDER BY 2 DESC, 1;
```

Why this matters: Snowflake documents that `CREDITS_ATTRIBUTED_COMPUTE` excludes idle time, so the residual bucket is expected to exist and should be surfaced explicitly (not hidden as “missing data”). [docs](https://docs.snowflake.com/en/sql-reference/account-usage/query_attribution_history)
